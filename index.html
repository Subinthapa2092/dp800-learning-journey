01

Introduction to Database and Database Users

2 hrs · What a database/DBMS is, database approach, actors, architecture

Assignment: Define data, database, DBMS, database system, database catalog Assignment Q ▸
  • Data: raw facts and figures with no particular meaning on their own (e.g. "John", "25", "Kathmandu").
  • Database: a shared, integrated collection of logically related data designed to meet the information needs of an organization.
  • DBMS: a computerized system software that lets users define, create, maintain, and control access to a database (examples: MySQL, Oracle, PostgreSQL, MongoDB, SQL Server).
  • Database system: the DBMS + the database + application programs + the users, working together as a whole.
  • Database catalog: the system-level store of metadata, the database's own self-description (table structures, data types, constraints) that the DBMS consults before executing operations.
Assignment: What is a DBMS? Architecture diagram & data independence Assignment Q ▸

DBMS: software that bridges the gap between raw stored data and meaningful information, while ensuring efficiency, security, and integrity of that data.

Database System Environment has five components you can draw as a diagram:

SoftwareDBMS + OS + network softwareMySQL, Windows/Linux
DataStored raw facts + metadataCustomer table, index files
ProceduresInstructions for using the DBMSBackup policy, login protocol
PeopleAll users involvedDBA, developers, end-users

Data independence: the ability to change the schema at one level of the database without having to change the schema at the next higher level: this is exactly what the three-schema architecture (Unit 2) is built to provide.

Assignment: Database schema & functions of a Database Administrator Assignment Q ▸

Database schema: the overall logical structure/description of the database, the "blueprint" (e.g. "Student has attributes RollNo: int, Name: string"). It's relatively static, unlike the actual data (instance), which changes constantly.

Functions of a DBA: authorizing access to the database, tuning performance, taking backups, handling recovery after failure, and defining/maintaining the schema.

Assignment: Centralized vs Client/Server architecture (with example) Assignment Q ▸

Centralized architecture: mainframe computers handle all processing: user applications, the interface, and DBMS functions all run in one location. Terminals just send keystrokes and receive screen output. Single point of control and failure.

Client/Server architecture splits work across tiers:

TierDescriptionExample
1-TierClient, server, and database all in one application, no network neededMS Access
2-TierClient application talks directly to the database server, via ODBC/JDBCLibrary management system in a school
3-TierClient → application server → database server; the client never talks to the DB directlyE-commerce store (checks stock, applies discounts, then hits the DB)
Assignment: Flat-file system & advantages of the DBMS approach Assignment Q ▸

Flat-file system: a data storage method where each application maintains its own separate data files (often plain text or simple records) with no central management, no relationships between files, and no shared query language, the traditional pre-DBMS approach.

Assignment: Data abstraction, data model, schemas, instances, database state Assignment Q ▸
  • Data abstraction: hiding the complex physical storage details from users, exposing only a simplified conceptual view.
  • Data model: a set of concepts/tools for describing data, relationships, constraints, and semantics of a database.
  • Schema: the static logical structure/description of the database (the "intension").
  • Instance: the actual data stored at a given moment (the "extension"): changes frequently.
  • Database state: the collection of all instances/data in the database at a particular point in time; every update moves the database from one valid state to another.

Database System Concepts

4 hrs · Data models, three-schema architecture, DB languages, interfaces, classification

Data models, schema vs instance Guaranteed ▸

Data model: a set of concepts/tools to describe data, relationships, constraints, and semantics. Types: high-level/conceptual (e.g. ER model), low-level/physical (record formats, pointers), representational/implementation (relational, network, object-oriented).

FeatureSchemaInstance
NatureDefinition / blueprintData / state
Change frequencyRareFrequent (insert/update/delete)
AnalogyA Java classA Java object
ANSI/SPARC three-schema architecture (with diagram description) Guaranteed ▸

The Three-Schema Architecture, proposed by the ANSI/SPARC committee, separates a database into three layers to achieve flexibility, abstraction, and data independence. When asked to draw it, sketch three stacked boxes labelled External → Conceptual → Internal, top to bottom.

  1. External Schema (View level): tailored views for different user groups; hides irrelevant details. Example: a customer sees only their balance, while a manager sees full financial reports.
  2. Conceptual Schema (Logical level): the single logical structure of the entire database, hiding physical storage. Example: defines entities like Customer and Account with attributes/relationships.
  3. Internal Schema (Physical level): the actual physical storage: file structures, indexes, compression, hashing.

Data independence it provides:

  • Logical independence: changes to the conceptual schema don't affect external schemas (e.g. adding a DOB column doesn't break users who don't need it): harder to achieve.
  • Physical independence: changes to the internal schema don't affect the conceptual schema (e.g. switching from sequential file to B+ tree index: users still query the same way): easier to achieve.

DBMS interfaces Common ▸
  • Menu-based: guided options for web/browsing users (dropdown menus)
  • Forms-based: users fill a form to input/request data
  • GUI: visual diagrams of tables/relationships, click-and-drag interaction
  • Natural language: users type/ask questions in plain language, system interprets into a query
  • Speech input/output: hands-free voice interaction
  • DBA interfaces: specialized privileged commands for admin tasks (creating accounts, tuning settings)
Classification of DBMS Guaranteed ▸

By data model: Relational (MySQL, Oracle), Object-Oriented (db4o), Hierarchical (IBM IMS), Network (IDMS), NoSQL (document: MongoDB, key-value: Redis, column-family: Cassandra, graph: Neo4j).

By user support: Single-user (SQLite for writing) vs Multi-user (PostgreSQL).

By distribution: Centralized (one server) vs Distributed/DDBMS (data spread across locations: homogeneous = same DBMS everywhere, heterogeneous = mixed DBMS software with middleware).

By cost: Open-source (MySQL, PostgreSQL) vs Commercial (Oracle).

By purpose: General-purpose (SQL Server) vs Special-purpose (optimized for streaming/real-time data).

Data Modelling using ER Model

7 hrs · Heaviest unit: ER diagrams, attributes, keys, cardinality, generalization

Assignment: Entity type/entity set + ER diagram for an airline ticket booking system Assignment Q ▸

Entity type: a category/template describing a group of similar objects (e.g. "Customer"). Entity set: the actual collection of entities of that type at a moment in time (e.g. all current customers).

Airline ticket booking ER sketch: entities and attributes to draw:

  • Customer(cust_id [PK], name, visit_count): visit_count supports the "filter frequent visitors" requirement
  • Ticket(ticket_no [PK], flight_no, price, booking_date)
  • Flight(flight_no [PK], source, destination, departure_time)
  • Discount(discount_id [PK], min_tickets, percentage): used to apply bulk-buy discounts
  • Relationships: Customer Books Ticket (1:N, meaning one customer books many tickets), Ticket For Flight (N:1), Customer Eligible For Discount (based on ticket count)

Draw each entity as a rectangle, attributes as ovals connected to their entity, and relationships as diamonds connecting the related entities, with cardinality labels (1, N) on the connecting lines.

Assignment: Referential integrity: what and why Assignment Q ▸

Referential integrity: a constraint enforced when a foreign key references the primary key of another relation: every foreign key value must either match an existing primary key value in the referenced table, or be NULL.

Why needed: it prevents "orphan" records: e.g. a Student row pointing to a Department that doesn't exist: keeping relationships between tables logically consistent. Full detail (violation causes, cascade/restrict/set-null handling) is covered under Unit 4.

Assignment: Relational database, characteristics of a relation, domain constraint Assignment Q ▸

Relational database: a database organized as a collection of relations (tables), each with rows (tuples) and columns (attributes), where relationships between tables are represented through shared key values rather than physical pointers.

Characteristics of a relation: no duplicate tuples, tuples are unordered, attributes are unordered, all values are atomic (1NF), and every attribute has a distinct name.

Domain constraint: specifies that every value taken by an attribute must come from that attribute's defined domain (set of atomic allowable values), e.g. an Age column must contain integers, not text.

Assignment: ER diagram for online course registration Assignment Q ▸
  • Student(student_id [PK], name, email)
  • Course(course_id [PK], title, credit_hours)
  • Registration: a relationship (many-to-many) between Student and Course, carrying its own attribute registration_date, since a student registers for many courses and a course has many students

Draw Student and Course as rectangles, "Registers" as a diamond between them with an M:N cardinality label, and registration_date as an attribute attached to the relationship diamond itself (since it depends on the student-course pair, not on either entity alone).

Assignment: Components of ER diagram, symbols, ER diagram for a library Assignment Q ▸

An ER diagram has three main components:

  1. Entity sets: rectangle (single = strong entity with its own PK, double rectangle = weak entity with no PK of its own)
  2. Attributes: ovals attached to entities (see attribute types card below)
  3. Relationship sets: diamond (single = strong relationship between two strong entities, double diamond = identifying relationship involving a weak entity)

Library ER sketch:

  • Book(isbn [PK], title, author, copies_available)
  • Member(member_id [PK], name, contact)
  • Issue: relationship between Book and Member with attributes issue_date, due_date (M:N: a member can issue many books over time, a book can be issued to many members over time)
Assignment: Integrity and its types Assignment Q ▸

Integrity: the overall correctness, consistency, and accuracy of data in a database, enforced through rules/constraints the DBMS checks automatically.

Types: domain integrity (values must come from the correct domain), entity integrity (no primary key attribute can be NULL), referential integrity (foreign keys must match an existing primary key or be NULL), key integrity/uniqueness (primary key values must be unique), and user-defined/semantic integrity (business rules, e.g. salary must be positive).

Concept: Attribute types (simple, composite, single/multi-valued, derived, key) Slide concept ▸
  • Simple: can't be divided further (e.g. Age)
  • Composite: made of smaller sub-attributes (e.g. Name → First + Last)
  • Single-valued: holds exactly one value per entity (e.g. DOB)
  • Multi-valued: can hold several values per entity (e.g. Phone_no, Email: drawn with a double oval)
  • Derived: computed from another attribute (e.g. Age derived from DOB: drawn with a dashed oval)
  • Key attribute: uniquely identifies each entity in the set (e.g. Roll_no)
Concept: Participation constraints, cardinality ratios, relationship degree Slide concept ▸

Participation constraint, the minimum number of relationship instances an entity must take part in: partial (single line, meaning participation is optional) vs total (double line, meaning every entity instance must participate).

Cardinality ratio: the maximum number of relationship instances an entity can have:

RatioMeaningExample
1:1At most one on each sidePerson: Passport
1:NOne on one side, many on the otherDepartment: Employees
N:1Many on one side, one on the other (same as 1:N, reversed direction)Employees: Department
M:NMany on both sidesStudent: Course

Degree of a relationship set = number of entity sets participating: unary (1: e.g. "married to" within Person), binary (2: most common, e.g. Student enrolled in Course), ternary (3), n-ary (n entities).

Concept: Specialization and Generalization Slide concept ▸

Generalization: forming a generalized superclass by extracting common characteristics from two or more classes (bottom-up). Example: Car and Motorcycle → generalized to Vehicle.

Specialization: the reverse: dividing a superclass into subclasses based on distinguishing characteristics (top-down). Example: Employee → Engineer, Manager, Technician.

Concept: Strong vs weak entity sets, strong vs weak relationship sets Slide concept ▸

Strong entity set: has its own primary key; drawn as a single rectangle. Weak entity set: has no primary key of its own and depends on a strong (owner) entity for identification; drawn as a double rectangle.

Strong relationship set: connects two strong entity sets; drawn as a single diamond. Weak/identifying relationship set: connects a strong entity to a weak entity; drawn as a double diamond.

Relational Data Model and Constraints

TermDefinition
RelationA table with rows and columns

AttributeA column in a relation (one property)

Characteristics of relations: no duplicate tuples, tuples are unordered, attributes are unordered, all values are atomic (no nested relations: 1NF), and every attribute has a distinct name.

Types of relational constraints (domain, tuple uniqueness, key, entity integrity, referential integrity) Guaranteed ▸
ConstraintRule
Domain constraintEach attribute's value must come from its defined domain (e.g. Age must be an integer)
Tuple uniquenessNo two tuples in a relation may be identical
Key constraintPrimary key values must be unique and never null
Entity integrityNo attribute that is part of the primary key can be null (applies to every component of a composite key too)
Referential integrityForeign key values must match an existing primary key value in the referenced relation, or be null
Keys in DBMS: all 10 types Guaranteed ▸

A key is a set of attributes that can uniquely identify each tuple in a relation.

KeyMeaning
Super keyAny set of attributes that uniquely identifies a tuple (no restriction on size)
Candidate keyA minimal super key: remove any attribute and it stops being unique
Primary keyThe candidate key chosen by the designer; never null, never changes, only one per relation
Alternate keyCandidate keys not chosen as the primary key
Foreign keyAn attribute whose values must match the primary key values of another (or the same) relation
Partial keyIdentifies a bunch of related tuples but not each one uniquely (used with weak entities)
Composite keyA primary key made of more than one attribute
Unique keyUnique across all rows, non-updatable, but may be null
Surrogate keyUnique, updatable, and can never be null (often an artificial ID)
Secondary keyUsed purely for faster indexing/searching, not for uniqueness
Referential integrity violations & how they're handled Guaranteed ▸

Three causes of violation:

  1. Insertion into the referencing relation: inserting a foreign key value that doesn't exist in the referenced relation's primary key
  2. Deletion from the referenced relation: deleting a row whose key value is still used by the referencing relation
  3. Update of the referenced relation: changing a primary key value that's still referenced elsewhere

For deletion/update, three handling strategies:

  • Cascade: automatically delete/update the matching rows in the referencing relation too
  • Restrict: reject/abort the delete or update if it's still referenced
  • Set null: set the referencing foreign key to NULL instead
Null value constraints Common ▸

A null represents the absence of a value: it is not the same as zero, blank, or an empty string. It can mean the value is unknown (exists but not recorded yet) or not applicable (doesn't make sense for this tuple, e.g. SpouseName for an unmarried person).

CREATE TABLE demo_student (
  StudentID INTEGER NOT NULL,
  Name TEXT NOT NULL,
  Email TEXT               -- NULL allowed here
);
INSERT INTO demo_student VALUES (1, 'Ram', NULL);
SELECT * FROM demo_student;
Relation schema, relational database, relational database schema Quick ▸
  • Relation schema: the logical definition of one relation: name + attributes + constraints. e.g. Student(StudentID: INT, Name: VARCHAR(50), Major: VARCHAR(4))
  • Relational database: a collection of normalized relations with distinct names, e.g. {Student, Course, Enrollment}.
  • Relational database schema: the set of all relation schemas plus all the integrity constraints across them.

06

Functional Dependencies and Normalization

6 hrs · Anomalies, functional dependencies, 1NF–4NF, decomposition properties

Assignment: Informal design guidelines for relational schemas Assignment Q ▸
  1. Semantic clarity: each tuple should represent one real-world entity/relationship: don't mix unrelated entities in one relation.
  2. Reduce redundancy: minimize duplicate data to save space and avoid inconsistencies.
  3. Simplicity: keep the number of attributes per relation manageable.
  4. Avoid nulls: minimize null values, especially in foreign keys/mandatory fields: lots of nulls usually means the attribute belongs in a different relation.
Assignment: Why functional dependencies? Explain 2NF and 3NF with example Assignment Q ▸

Why FDs matter: a functional dependency X → Y means that if two tuples agree on X, they must also agree on Y: "X determines Y". FDs are the formal foundation for detecting redundancy and deciding how to normalize a relation, since normal forms are literally defined in terms of which FDs are/aren't allowed.

2NF: a relation is in 2NF if it's already in 1NF and has no partial dependency: no non-prime attribute depends on only part of a composite candidate key.

-- Example: StudentCourse(StudentID, CourseID, StudentName, CourseName, Grade)
-- Primary key = (StudentID, CourseID)
-- StudentName depends only on StudentID, CourseName only on CourseID -> partial dependency, violates 2NF

-- Fix: split into three 2NF tables
CREATE TABLE student(StudentID INTEGER PRIMARY KEY, StudentName TEXT);
CREATE TABLE course(CourseID INTEGER PRIMARY KEY, CourseName TEXT);
CREATE TABLE enrollment(StudentID INTEGER, CourseID INTEGER, Grade TEXT,
  PRIMARY KEY(StudentID, CourseID));

INSERT INTO student VALUES (1,'Ram'),(2,'Sita');
INSERT INTO course VALUES (10,'DBMS'),(20,'Python');
INSERT INTO enrollment VALUES (1,10,'A'),(1,20,'B+'),(2,10,'A+');

SELECT s.StudentName, c.CourseName, e.Grade
FROM enrollment e
JOIN student s ON s.StudentID = e.StudentID
JOIN course c ON c.CourseID = e.CourseID;

3NF: a relation is in 3NF if it's already in 2NF and has no transitive dependency for non-prime attributes: no non-prime attribute depends on another non-prime attribute. Example: StaffNo → BranchNo and BranchNo → BranchAddress means StaffNo → BranchAddress transitively; fix by moving BranchAddress into a separate Branch table keyed on BranchNo.

Assignment: Benefits of using normalization Assignment Q ▸
  • Eliminates data redundancy, saving storage space
  • Prevents insertion, update, and deletion anomalies
  • Improves data consistency and integrity
  • Makes the database easier to maintain and extend
  • Produces a cleaner, more logical schema aligned with real-world entities
Concept: Anomalies: insertion, update, deletion Slide concept ▸

Anomalies happen when a poorly normalized table forces inconsistent or incomplete data handling.

  • Insertion anomaly: can't add a valid new fact because unrelated data is also required. Example: can't add a new course with no students yet, because the table also demands student info.
  • Update anomaly: the same fact is stored in multiple rows, so an update must touch every row: miss one and the data becomes inconsistent (e.g. two different instructors shown for the same course).
  • Deletion anomaly: deleting one fact accidentally destroys another. Example: deleting the last student enrolled in a course also wipes out the course's own details.

Causes: data redundancy, lack of normalization, poor design (unrelated entities combined), missing relationships/foreign keys, and partial/transitive dependencies.

Concept: Normal forms: 1NF, 2NF, 3NF, BCNF, 4NF Slide concept ▸
FormRule
1NFEvery cell holds a single atomic value (no repeating groups/multi-valued cells)
2NF1NF + no partial dependency (non-prime attribute depending on only part of a composite key)
3NF2NF + no transitive dependency for non-prime attributes
BCNF3NF + for every non-trivial FD A→B, A must be a super key (stricter than 3NF)
4NFBCNF + no non-trivial multivalued dependencies (e.g. splitting a user's multiple phone numbers and multiple emails into separate tables instead of one combined table)
Concept: Full, partial, and transitive functional dependency Slide concept ▸
  • Full functional dependency: B depends on the entire candidate key, not just part of it. Example: in StudentCourse(StudentID, CourseID → Grade), Grade genuinely needs both attributes together.
  • Partial dependency: a non-prime attribute depends on only part of a composite candidate key. Example: in (StaffNo, StaffName) → BranchNo, StaffNo alone already determines BranchNo, so StaffName wasn't needed: that's partial.
  • Transitive dependency: A → B → C implies A → C indirectly, where A is not a super key and C is non-prime. Example: StaffNo → BranchNo and BranchNo → BranchAddress gives StaffNo → BranchAddress transitively.
Concept: Decomposition properties: lossless join & dependency preservation Slide concept ▸

Decomposition: splitting one relation into two or more sub-relations. Every decomposition should ideally satisfy:

  • Lossless join: joining the sub-relations back together (natural join) reproduces exactly the original relation, with no missing or extra (spurious) tuples. Condition: R1 ⋈ R2 = R.
  • Dependency preservation: none of the original relation's functional dependencies are lost: each sub-relation still enforces the FDs that applied to it in the original.

A decomposition that fails the lossless condition (R1 ⋈ R2 ⊃ R, producing extra rows) is called a lossy join decomposition and introduces spurious tuples: this is exactly what normalization tries to avoid when splitting tables.
